S3 Mi Cuenta crash
.select() omitía items → TypeError		 Mongoose omite campos silenciosamente — el frontend asumía que el campo existía Toda ruta con .select() debe tener test que valide qué campos retorna
S4 Cart merge 401
Retry interceptor limpiaba auth si merge sin token fallaba		 Efectos secundarios no documentados en el interceptor de Axios Los interceptores de Axios son globales — documentar y testear sus efectos secundarios

Handler POST, Transbank redirige con GET + query string		 La documentación de Transbank describe parámetros pero el método HTTP estaba en otra sección Leer el diagrama de flujo completo del gateway, no solo la referencia de parámetros

06 — Decisiones técnicas clave
ARCH
Dos repositorios separados (no monorepo)
Simplifica el deploy independiente en Railway. Alineado con cliente single-tenant de un local. Contrapartida: CORS multi-origin requiere configuración explícita desde el inicio.
ARCH
Snapshot pattern en order items
Los ítems de pedido copian nombre, precio y SKU del producto al momento de la compra. Un cambio de precio posterior no afecta pedidos históricos. Es el patrón correcto para e-commerce.
ARCH
Transformer pattern (toFrontendOrder)
Separa el contrato de la API del tipo interno del frontend. Centraliza el mapeo en un solo lugar — cada bug de transformación se corrige en un punto y se protege con un test.
TEST
MongoMemoryServer para todos los tests
Tests de integración reales sin depender de Atlas. Cada test levanta su propia BD en memoria — aislamiento total, sin flakiness. Crecimiento: 20→27→34→40→46 tests a lo largo de los sprints.
DEV
VITE_USE_MOCKS para desacoplar desarrollo
El frontend puede desarrollarse sin backend activo. Los mocks simulan datos realistas. Permite trabajo paralelo en frontend/backend sin bloqueos de API.
SEC
Zustand con sessionStorage para auth (no localStorage)
El token no sobrevive al cierre del navegador — sesión más segura por defecto. El usuario puede optar por "recordarme" si se implementa en el futuro.
RISK
express-rate-limit instalado pero no activado
El paquete está en package.json desde Sprint 2 pero nunca se configuró en las rutas de auth. Es deuda técnica P1 — fácil de activar (2h), pero representa un riesgo de brute force en producción.
07 — Lo que salió bien (replicar)
TypeScript end-to-end desde el primer commit
Evitó errores de contrato entre repos. Los tipos de Order, Product y User son consistentes en ambos lados. El compilador detectó múltiples errores que habrían llegado a producción.
Smoke test con pedido real como ceremonia de cierre de sprint
Post Sprint 4: 3 bugs detectados. Post Sprint 6: 1 bug (WebPay GET). Sin este protocolo, todos habrían llegado al primer cliente. El smoke test es barato y tiene ROI altísimo.
Separación service / controller / route
Los servicios de pago (mercadopago.service.ts, webpay.service.ts) son testeables de forma independiente. Mockear un servicio en un test es trivial. La lógica de negocio no está acoplada a Express.
VITE_USE_MOCKS como interruptor de desarrollo
El frontend avanzó varios sprints sin depender del estado del backend. Los mocks con datos realistas aceleran el desarrollo de componentes UI y permiten demostrar funcionalidades antes de que el endpoint exista.
Scope creep positivo gestionado
CRUD Direcciones, Analytics, Reset Password y OG image no estaban en el plan original. Se incorporaron sin desestabilizar los sprints y elevaron significativamente el valor percibido del producto.
Registro formal de deuda técnica priorizada
13 ítems con prioridad P0–P3, esfuerzo estimado y descripción del fix. Permite al equipo decidir qué atacar primero sin depender de memoria ni de los commits. Es un artefacto transferible.
08 — Lo que mejorar (evitar en próximos proyectos)
El bug del .select() ocurrió dos veces (Sprint 3 y Sprint 9)
Problema: Mongoose omite campos silenciosamente cuando se usa .select(). El frontend recibía undefined en lugar de un error explícito. Fix sistémico: toda ruta con .select() debe tener un test que valide cada campo en la respuesta — no solo el status code.
imageUrl base64 en order items llegó a producción
Problema: Los ítems de pedido guardan la imagen como base64 completo (300–600 KB por producto). Un pedido con 4 items puede pesar 1+ MB. Causa: el código de base64 se usó como placeholder antes de Cloudinary y nunca se limpió. Fix futuro: guardar solo la URL de Cloudinary desde el inicio.
Rate limiting instalado pero no activado
Problema: express-rate-limit está en package.json desde Sprint 2 pero nunca se configuró. Los endpoints de auth (/login, /forgot-password) son vulnerables a brute force. Fix futuro: activar rate limiting en las rutas de auth como parte del checklist del Sprint 1, no del backlog.
0% cobertura de tests en frontend
Problema: El bug de total: raw.total (que causó $NaN en producción) no tiene test. Si se hubiera testeado toFrontendOrder(), se habría detectado al escribir el test. Fix futuro: al menos testear las funciones de transformación de datos — son donde más bugs se esconden.
.env.example desactualizado varios sprints
Las variables de Cloudinary, Resend, MercadoPago y WebPay se agregaron a Railway de forma manual pero el archivo .env.example no se actualizó progresivamente. Al final del proyecto había más variables en Railway que en el ejemplo. Fix futuro: actualizar el archivo en el mismo commit que agrega la variable.
Textos legales entregados como placeholder
Las 3 páginas legales (términos, privacidad, política de envíos) se generaron con estructura pero sin contenido final. El cliente no los completó antes del cierre del MVP. Fix futuro: solicitar los textos legales en Sprint 1 con un deadline explícito — son un bloqueante para la primera venta.
09 — Lecciones aprendidas por categoría
Arquitectura
Snapshot pattern en orders es la decisión correcta para e-commerce — el precio del producto puede cambiar después de la compra.
Dos repos separados simplifican Railway pero complican CORS en desarrollo. Documentar CORS multi-origin desde Sprint 1.
Aggregation pipelines con $lookup anidado requieren preserveNullAndEmptyArrays: true para no perder registros sin categoría.
Desarrollo
Toda ruta con .select() debe tener test que valide los campos retornados — no solo el status 200.
Al mapear documentos Mongoose, extraer campos explícitamente. El spread ({ ...doc }) incluye metadata interna.
Los gateways de pago tienen restricciones distintas en sandbox vs producción. Documentarlas antes de empezar el sprint.
Testing
MongoMemoryServer es la estrategia correcta — no mockear el ORM, testear contra una BD real en memoria.
Los tests deben validar el contrato completo de la API: campos específicos, tipos y valores — no solo el status code.
Al menos testear las funciones de transformación de datos en el frontend (como toFrontendOrder) — es donde más bugs se acumulan.
Deploy
Railway auto-deploy desde main es el flujo óptimo para sprints cortos — cero fricción en el CI.
Smoke test con pedido real debe ser la ceremonia de cierre de cada sprint, no opcional.
Actualizar .env.example en el mismo commit que agrega una variable de entorno — no al final del proyecto.
Gestión de proyecto
Scope creep positivo: las funcionalidades no planificadas que elevan el valor percibido son bienvenidas si no desestabilizan el sprint. Identificarlas y decidir explícitamente antes de implementar. Propuesta comercial en paralelo al MVP: construir el argumento de venta mientras se desarrolla el producto, no al final. Evita la disonancia entre lo que se entrega y lo que se vende.
Registrar deuda técnica formalmente con prioridad y esfuerzo estimado — es más útil que mencionarla en los commits y olvidarla. Solicitar entregables del cliente (textos legales, teléfono, logo) en Sprint 1 con deadline explícito — los placeholders que llegan a producción erosionan la confianza.
